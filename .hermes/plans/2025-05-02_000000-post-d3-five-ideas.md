# Plan: Post-D3 Cleanup & Feature Batch

> **Goal:** Implement all 5 accepted ideas from `ideate-2025-05-02-spall-post-d3.md`.
> **Order:** Hygiene first (clean base), then self-contained units (tests, history search), then UX polish (REPL errors), finally the structural refactor (`ResponseContext`).
> **Assumption:** The D-series branch (D1, D2, D3) is merged or stashed; all builds pass. No new dependencies needed beyond those already in `Cargo.lock`.

## Phase 1 — Hygiene: Remove global `unused = "allow"` (XS effort)

**Objective:** Stop suppressing dead-code warnings so the compiler surfaces real issues.

**Files:**
- `spall-cli/Cargo.toml`
- `spall-cli/src/commands/mod.rs`
- `spall-cli/src/commands/api.rs`
- `spall-cli/src/commands/history.rs`
- `spall-cli/src/auth/oauth2.rs`
- Any other file that starts throwing warnings after removal

**Steps:**

1. **Delete the global lint override**
   ```diff
   - [lints.rust]
   - unused = "allow"
   ```
2. **Delete per-file overrides**
   Remove `#![allow(dead_code, unused_variables, unused_imports)]` from `commands/mod.rs` and `#![allow(unused_imports)]` from `commands/api.rs`.
3. **Compile and fix**
   ```bash
   cargo build --workspace 2>&1 | tee /tmp/build.log
   ```
   For each warning, either:
   - `cargo fix --workspace` for mechanical fixes, or
   - manual removal if it's truly dead code (e.g., `oauth2.rs` stubs).
4. **Verify**
   ```bash
   cargo clippy --workspace -- -D warnings
   cargo test --workspace
   ```

**Risks:** Some warnings may be from intentionally-unimplemented stubs (OAuth2, daemon, etc.). For those, use targeted `#[allow(dead_code)]` or `todo!()` comments rather than a global blanket.

## Phase 3 — DX/UX: `history search` subcommand (S effort)

**Files:**

- `spall-cli/src/commands/history.rs` — wire `search` subcommand
- `spall-cli/src/main.rs` — add clap args for `history search`

**Database query:**
```sql
SELECT id, timestamp, api, operation, method, url, status_code, duration_ms
FROM requests
WHERE (?1 IS NULL OR api LIKE ?1)
  AND (?2 IS NULL OR status_code = ?2)
  AND (?3 IS NULL OR method LIKE ?3)
  AND (?4 IS NULL OR url LIKE ?4)
  AND (?5 IS NULL OR timestamp >= ?5)
ORDER BY timestamp DESC
LIMIT ?6
```

**API signature:**
```rust
// history.rs
pub fn search(
    &self,
    api_pattern: Option<&str>,

    limit: usize,

```

**CLI surface (`main.rs`):**

```
spall history search

    [--status <code>]
    [--method <GET|POST|... >]
    [--url <pattern>]
    [--since "2025-01-01"]

```

**Steps:**

## Phase 4 — DX/UX: Structured REPL pipe error reporting (S effort)

**Objective:** Replace `eprintln!("Pipe error: {:?}", e)` with per-stage, actionable diagnostics.

**Files:**
- `spall-cli/src/repl.rs`
- `spall-cli/src/chain.rs` (minor: propagate span context)

**Design:**

Introduce a small error type in `repl.rs`:

```rust
#[derive(Debug)]
enum PipeError {
    Parse { stage: usize, expr: String, reason: String },
    JmesPath { stage: usize, expr: String, reason: String },
    NoPreviousResponse { stage: usize },
    Http { stage: usize, source: crate::SpallCliError },
}
```

In `run_piped`, wrap each stage failure:

```rust
for (stage_idx, stage) in stages.iter().enumerate() {
    if stage_idx == 0 { ... } else {
        let prev = take_last_response()
            .ok_or(PipeError::NoPreviousResponse { stage: stage_idx })?;
        let chain = ChainExpr::parse(stage)
            .map_err(|e| PipeError::Parse { stage: stage_idx, expr: stage.to_string(), reason: e.to_string() })?;
        let args = chain.resolve(&prev)
            .map_err(|e| PipeError::JmesPath { stage: stage_idx, expr: stage.to_string(), reason: e.to_string() })?;
        // ... run_with_args
    }
}
```

**Pretty-print:**
```
Pipe failed at stage 2: JMESPath error
  Expression: $.data.id
  Reason:     Invalid expression
  
  To debug:   echo '{"data":{}}' | spall chain validate '$.data.id'
```

**Steps:**
1. Define `PipeError` enum in `repl.rs`.
2. Update `run_piped` to return `Result<(), PipeError>` and map each failure site.
3. Add `Display` impl for `PipeError` (or use `miette` diagnostic derive).
4. Update REPL loop to print `pipe_error.to_string()` instead of `{:?}`.
5. Manual test: launch REPL, trigger a bad pipe `petstore get-pet-by-id 1 | badop --id $.nonexistent`.
6. Commit.

## Phase 5 — Refactoring: Replace `LAST_RESPONSE` static with `ResponseContext` (M effort)

**Objective:** Make response passing explicit and testable, eliminating the global mutable singleton.

**Files:**
- `spall-cli/src/execute.rs` — introduce `ResponseContext`, remove `static LAST_RESPONSE`
- `spall-cli/src/main.rs` — thread `ResponseContext` through `run_with_args` and `handle_api_operation`
- `spall-cli/src/repl.rs` — create `ResponseContext` at pipe entry, pass by mutable reference

**New type:**
```rust
// spall-cli/src/execute.rs
#[derive(Debug, Default)]
pub struct ResponseContext {
    last: Option<serde_json::Value>,
}

impl ResponseContext {
    pub fn store(&mut self, value: serde_json::Value) {
        self.last = Some(value);
    }
    pub fn take(&mut self) -> Option<serde_json::Value> {
        self.last.take()
    }
}
```

**Signature changes:**

```rust
// execute.rs
pub async fn execute_operation(
    op: &ResolvedOperation,
    ...
    ctx: &mut ResponseContext,
) -> Result<OperationResult, SpallCliError> {
    ...
    ctx.store(value.clone());
    Ok(OperationResult { status, value })
}

// main.rs
pub async fn run_with_args(
    args: &[String],
    registry: &ApiRegistry,
    cache_dir: &Path,
    ctx: &mut ResponseContext,
) -> miette::Result<()> { ... }

async fn handle_api_operation(
    ...
    ctx: &mut ResponseContext,
) -> miette::Result<()> { ... }
```

**REPL pipe integration:**
`run_piped` creates a local `ResponseContext::default()`, passes `&mut ctx` into each `run_with_args` call. After the pipe finishes (or errors), the context is dropped — no stale data leaks to the next REPL command.

**Steps:**
1. Add `ResponseContext` to `execute.rs`, delete `static LAST_RESPONSE`, delete `store_last_response` / `take_last_response`.
2. Update `execute_operation` to accept `ctx: &mut ResponseContext`.
3. Update `handle_api_operation` signature and all call sites in `main.rs`.
4. Update `run_with_args` signature and the `Box::pin` call site in `repl.rs`.
5. Update `run_piped` to own a `ResponseContext` and pass it.
6. Build: `cargo build --workspace`.
7. Test: `cargo test --workspace`.
   - If any existing test inspected `LAST_RESPONSE`, convert it to pass a `ResponseContext`.
8. Commit.

**Rollback plan:** If signature threading proves too invasive due to `Box::pin` recursion (type inference changes), keep `ResponseContext` as a module-local wrapper but **retain the static** temporarily, marked with a `// TODO: eliminate static after async call signatures stabilize`. This gives 80% of the win (testability) without blocking the batch.

## Risks, Trade-offs, and Open Questions

- **Phase 5 risk:** `ResponseContext` threading may conflict with the `Box::pin` async recursion pattern. If `Box::pin(run_with_args(..., ctx)).await` fails to compile due to lifetime inference, we may need to wrap `ctx` in an `Arc<Mutex<...>>` and clone the Arc into each future. This is slightly more complex but still testable and avoids globals.
- **Phase 3 scope:** Does `--since` accept ISO-8601, epoch seconds, or human strings like "1h ago"? **Decision:** ISO-8601 `YYYY-MM-DD` only for this batch. Extend later.
- **Phase 4 scope:** Should pipe errors be logged to the history database? **Decision:** No, history records complete requests, not partial pipe failures. Keep history schema unchanged.
- **Parallelism:** Phases 2, 3, and 4 are independent once Phase 1 is done. They can be worked in parallel (e.g., via subagents) or sequentially. Phase 5 should be last because it changes the most signatures.
